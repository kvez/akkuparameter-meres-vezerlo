Választás: A) QThread

A TestRunner.run() maradjon blokkoló, GUI-független üzleti logika.
A GUI-ban ezt egy külön QThread-ben kell futtatni, PySide6 signal/slot kommunikációval.

  - TestRunner nem hoz létre saját threadet
  - TestRunner nem importál PySide6-ot
  - TestRunner run() blokkoló
  - GUI worker threadből hívja
  - GUI nem fagy be
  - stop / emergency stop kívülről flaggelhető


MIÉRT QThread?
==============

1. PySide6 natív megoldás
-------------------------

Mivel a GUI PySide6 lesz, a QThread a természetes választás.

Előnyök:

  - signal/slot natív
  - GUI thread biztonságosan frissíthető signalokon keresztül
  - nem kell asyncio integráció
  - nem kell külön process IPC
  - jól illik a Qt alkalmazás életciklusához


2. A TestRunner tiszta marad
----------------------------

A TestRunner nem tud a GUI-ról.

Tehát ez TILOS a TestRunnerben:

  - PySide6 import
  - QThread létrehozás
  - QWidget hívás
  - signal emit közvetlenül

A GUI oldali Worker burkolja be:

  runner.run(test_plan)

A worker ad signalokat:

  sample_ready(dict)
  event_ready(dict)
  finished(TestResult)
  fault(str)


3. Egyszerű stop modell
-----------------------

GUI gombok:

  Stop:
    worker.runner.request_stop()

  Emergency Stop:
    worker.runner.request_emergency_stop("USER_EMERGENCY_STOP")

A TestRunner következő tickben feldolgozza.

Emergency stop esetén:

  - CHARGE/DISCHARGE közben is azonnal safe_off
  - RELAX/OCV alatt is leáll
  - GUI nem blokkol


MIÉRT NEM asyncio?
==================

A B) asyncio + qasync most nem kell.

Indok:

  - PyVISA / műszerkommunikáció jellemzően blokkoló
  - a TestRunner run() eleve blokkoló ciklus
  - qasync plusz komplexitás
  - több hibalehetőség
  - FÁZIS 6-ban nem ad elég előnyt

Asyncio akkor lenne érdekes, ha:

  - sok hálózati async I/O lenne
  - websocket / REST / több párhuzamos háttérfolyamat lenne
  - eleve async driver stack készült volna

Most nem ez a helyzet.


MIÉRT NEM subprocess?
=====================

A C) subprocess túl nagy lépés most.

Előnye lenne:

  - teljes izoláció
  - GUI crash és mérőmotor crash jobban szétválik
  - később akár headless mérőszolgáltatás is lehetne

Hátránya most:

  - IPC kell
  - parancsprotokoll kell
  - log/signal átadás bonyolultabb
  - instrument objektumok processhatáron nem vihetők át
  - debug nehezebb

Subprocess később jó lehet „labor daemon” irányba, de FÁZIS 6 MVP-hez nem kell.

class TestRunnerWorker(QObject):
    sample_ready = Signal(dict)
    event_ready = Signal(dict)
    finished = Signal(object)
    fault = Signal(str)

    def __init__(self, runner: TestRunner, test_plan: TestPlan):
        super().__init__()
        self.runner = runner
        self.test_plan = test_plan

    @Slot()
    def run(self):
        try:
            result = self.runner.run(self.test_plan)
            self.finished.emit(result)
        except Exception as exc:
            self.runner.request_emergency_stop(f"WORKER_EXCEPTION: {exc}")
            self.fault.emit(str(exc))

    @Slot()
    def request_stop(self):
        self.runner.request_stop()

    @Slot()
    def request_emergency_stop(self):
        self.runner.request_emergency_stop("USER_EMERGENCY_STOP")


FONTOS:
  A TestRunnerWorker lehet PySide6-os.
  A TestRunner ne legyen PySide6-os.

THREAD SAFETY SZABÁLYOK
=======================

1. GUI widgetet csak GUI threadből szabad frissíteni.

2. Worker thread csak signalokat emitáljon.

3. Műszereket csak a worker thread / TestRunner érje el.

4. GUI közvetlenül ne hívjon driver metódust.

5. Stop gomb csak flaget állítson:
     runner.request_stop()

6. Emergency Stop gomb:
     runner.request_emergency_stop("USER_EMERGENCY_STOP")

7. A runner.safe_all_off() a worker threadben fusson.

8. Ha emergency stop gombot megnyomják, a GUI azonnal váltson „Stopping...” állapotra,
   de a tényleges safe_off eredményt a worker event/final signal adja vissza.

MIÉRT NEM A?
============

A) Popup dialógus önmagában nem ideális.

Hátrányok:

  - eltakarhatja a grafikonokat
  - véletlenül bezárható
  - hosszú kézi munka alatt zavaró
  - nem alkalmas részletes checklist megjelenítésére
  - rosszabb laboros workflow

Popup csak jelzésnek jó, nem fő állapotkezelésnek.


MIÉRT NEM C?
============

C) Automatikus STOPPED státusz + újraindítás túl darabos.

Hátrány:

  - a felhasználónak újra kell indítania a következő lépést
  - nehezebb egyben kezelni a BQ learning sessiont
  - checkpoint után a folyamat folytathatósága kevésbé kontrollált
  - több hibalehetőség: rossz test_plan, rossz folytatási pont, új session indulhat

A runner persze belül használhat "WAITING_FOR_MANUAL_CHECKPOINT" vagy hasonló státuszt,
de GUI szinten legyen egyértelmű folytatás gomb.


MIÉRT NEM D?
============

D) Most nem kell tervezni — ezt nem választanám.

Indok:

  Már a TestRunner tervben szerepel a MANUAL_CHECKPOINT.
  Ha a GUI MVP-ben még csak CHARACTERIZATION fut, akkor is érdemes az állapotmodellt
  előre jól kialakítani.

Nem kell minden BQ részletet megvalósítani az első GUI-ban, de az architektúrában
legyen helye a checkpointnak.

    enabled


JAVASOLT RUNNER API
===================

A TestRunnerben később legyen:

  request_continue_from_checkpoint()

vagy általánosabban:

  request_continue()

A MANUAL_CHECKPOINT nem legyen sima DONE.

Lehetséges működés:

  _run_manual_checkpoint(step):
      safe_all_off()
      logger.log_event("MANUAL_BQ_CHECKPOINT_REACHED", ...)
      checkpoint mentés
      self.status = "WAITING_FOR_MANUAL_CHECKPOINT"

      while not self.continue_requested:
          if self.emergency_stop_requested:
              return self._emergency_stop(...)
          if self.stop_requested:
              return self._graceful_stop("USER_STOP_AT_MANUAL_CHECKPOINT")
          sleep vagy tick

      return TestResult(status="DONE")

MVP-ben egyszerűbb változat:

  - runner visszatér TestResult(status="STOPPED", reason="MANUAL_BQ_CHECKPOINT_REACHED")
  - GUI megjeleníti a checkpoint panelt
  - folytatás későbbi fázisban

De hosszú távon jobb:

  - runner várakozó állapotban marad
  - GUI Continue gombbal megy tovább


JAVASOLT CHECKPOINT PANEL TARTALOM
==================================

Cím:
  MANUAL BQ CHECKPOINT

Állapot:
  A fizikai BQ learning ciklus elérte a kézi ellenőrzési pontot.
  PSU OFF, Load OFF, aktív energiaút nincs.

Checklist:

  [ ] BQ eszköz csatlakoztatva
  [ ] bqStudio / BQ tool elindítva
  [ ] UpdateStatus ellenőrizve
  [ ] Qmax ellenőrizve / mentve
  [ ] Ra table ellenőrizve / mentve
  [ ] Log / export fájl elmentve
  [ ] Kezelő megerősíti, hogy a folyamat folytatható vagy lezárható

Gombok:

  - Checkpoint kész / Folytatás
  - Teszt lezárása
  - Jegyzet mentése
  - Emergency Stop
