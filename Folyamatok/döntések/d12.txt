RÉSZLETES INDOKLÁS
==================

1. Tab váltás "Folytatás" gombra kattintáskor
---------------------------------------------

Választás:

  B) Marad a Checkpoint tabon, csak status_changed("RUNNING") után vált

Indok:

A gombnyomás önmagában még nem bizonyítja, hogy a folytatás tényleg elindult.

Előtte még történhet:

  - konfigurációs hiba
  - start_step_index hiba
  - worker indítási hiba
  - precheck hiba
  - műszer reconnect / safe_all_off / check hiba
  - kivétel a runner indításakor

Ha azonnal átváltunk Live tabra, de a run nem indul el, a GUI félrevezető lehet.

Javasolt működés:

  Folytatás gomb megnyomása:
    - Checkpoint tab marad
    - status: "Folytatás indítása..."
    - gombok disabled

  Ha status_changed("RUNNING") érkezik:
    - átváltás Live tabra
    - LivePanel aktív
    - checkpoint panel inaktív

  Ha fault / hiba érkezik:
    - marad Checkpoint tabon vagy Fault státuszra vált
    - hibaüzenet látható

Ez determinisztikusabb és biztonságosabb.


2. LivePanel állapota folytatáskor
----------------------------------

Választás:

  A) reset_plots() hívódik

Indok:

A korábbi döntés szerint checkpoint utáni folytatás / új mérés új session szemantikájú:

  - Ah integrátor reset
  - új session fájlok
  - új mérési kontextus
  - parent_session_id kapcsolhatja az előzőhöz

Ezért a LivePanel grafikonok is induljanak tiszta állapotból.

Ha vizuálisan akkumulálnánk az előző adatokat, az félrevezető lenne:

  - új session fut, de régi grafikonon folytatódik
  - Ah resetel, de grafikon folytonosnak látszik
  - riport és live view eltérő szemantikát sugall

Javaslat:

  request_continue_from_checkpoint előtt vagy sikeres RUNNING után:

    live_panel.reset_plots()
    live_panel.clear_live_values()
    live_panel.clear_events() opcionális

Én az event logot nem feltétlen törölném automatikusan, vagy legalább írnék bele:

  "Új session indult checkpoint után."

Grafikon:
  reset

Event log:
  vagy reset + új session event,
  vagy marad, de legyen egy elválasztó sor.

MVP-hez:
  reset_plots() mindenképp.


3. Runner belső állapot flagek folytatás előtt
----------------------------------------------

Választás:

  A) Automatikus reset a Worker request_continue_from_checkpoint() belsejében

Indok:

Ez a legbiztonságosabb és leglokalizáltabb.

A MainWindow ne tudjon túl sokat a runner belső flagjeiről.
A worker az összekötő réteg, ezért a continuation indítás előtt ő végezze el a
futtatáshoz szükséges állapot-normalizálást.

Resetelendő:

  stop_requested = False
  emergency_stop_requested = False
  emergency_stop_reason = ""
  continue_requested = False, ha van ilyen
  esetleges checkpoint/finished ideiglenes state

Javasolt metódus:

  runner.reset_control_flags()

Vagy workerben:

  if hasattr(self.runner, "reset_control_flags"):
      self.runner.reset_control_flags()
  else:
      self.runner.stop_requested = False
      self.runner.emergency_stop_requested = False
      self.runner.emergency_stop_reason = ""

Jobb:
  legyen explicit TestRunner.reset_control_flags()

Indok:

  - egy helyen van a szabály
  - később bővíthető
  - MainWindow egyszerű marad
  - nem marad bent korábbi STOP flag


4. CheckpointPanel állapota folytatás után
------------------------------------------

Választás:

  B) A tab látható marad, de inaktív amíg a worker fut

Indok:

Nem érdemes eltüntetni / letiltani teljesen a tabot, mert a kezelőnek hasznos lehet
visszanézni:

  - milyen checkpoint volt
  - mit kellett ellenőrizni
  - milyen jegyzetet írt
  - mi volt a next_step_index
  - mi volt a session kapcsolat

De a gombokat inaktiválni kell, hogy ne lehessen duplán folytatni.

Javasolt állapot:

  CheckpointPanel látható marad
  Folytatás gomb disabled
  Teszt lezárása gomb disabled vagy csak safe állapotban enabled
  Note mező read-only vagy mentett állapotban
  státusz:
    "Folytatás elindítva — mérés fut."

Amikor status_changed("RUNNING") érkezik:

  MainWindow átvált Live tabra
  CheckpointPanel tab továbbra is elérhető, de inaktív

Ha a folytatás DONE/FAULT/STOPPED:

  checkpoint panel maradhat read-only


5. Ha "Folytatás" gombra kattintanak, de _checkpoint_next_step_index is None
----------------------------------------------------------------------------

Választás:

  B) Loggol egy WARNING-ot és nem csinál semmit

Indok:

Elvileg C igaz: ha a gomb csak resume_possible=True esetén enabled, akkor indexnek
kellene lennie.

De GUI-ban / állapotgépben jobb a defenzív viselkedés.

Ha valami mégis rossz:

  - ne induljon el hibásan
  - ne dobjon nyers kivételt a GUI threadben
  - ne fusson start_step_index=None értékkel
  - legyen nyoma az eseménylogban

Javasolt esemény:

  event_code: "CHECKPOINT_RESUME_INVALID_STATE"
  severity: "WARNING"
  event_message: "Folytatás kérése érkezett, de nincs érvényes next_step_index."

GUI:

  - marad Checkpoint tabon
  - status bar:
      "Folytatás nem indítható: hiányzó checkpoint index"
  - Folytatás gomb disabled


ÖSSZESÍTETT DÖNTÉS
==================

1. Tab váltás:
   B — csak RUNNING státusz után váltson Live tabra.

2. LivePanel:
   A — reset_plots(), mert új session / resetelt Ah szemantika.

3. Runner flags:
   A — worker resetelje automatikusan, lehetőleg runner.reset_control_flags() hívással.

4. CheckpointPanel:
   B — maradjon látható, de futás alatt inaktív.

5. Hiányzó next_step_index:
   B — WARNING log + no-op.

Javasolt implementációs sorrend
===============================

1. TestRunner.reset_control_flags() hozzáadása.
2. Worker.request_continue_from_checkpoint():
     - ellenőrzi next_step_index
     - ha None: event WARNING + return
     - runner.reset_control_flags()
     - új run indítása start_step_index-szel
3. MainWindow:
     - Folytatás gombnál még nem vált tabot
     - status_changed("RUNNING") esetén vált Live tabra
4. LivePanel:
     - reset_plots() folytatás indításakor vagy RUNNING előtt
5. CheckpointPanel:
     - set_running_after_checkpoint() / set_inactive_running()
